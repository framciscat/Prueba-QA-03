Ejecución de pruebas + Conclusiones y sugerencias

Microservicios API ARTIST

Ejecución plan de pruebas API REST (Semana 16 Octubre)

Se empleó el plan de pruebas propuesto para la aplicación “API REST ARTIST”. Para ello, se utilizaron pruebas manuales con el software “Postman” para observar el funcionamiento de los Endpoint de cada entidad. Para la redacción y registro de resultados de casos de prueba, se utilizó el software Test Link.

Se ejecutaron pruebas de tipo funcional, debido a que se necesitaba conocer el comportamiento de la aplicación frente a diversas modificaciones en sus datos de entrada, funcionamiento general de los métodos HTTP y algunos aspectos de seguridad como el acceso de datos.

Conclusiones:

El plan de prueba constó de un total de 12 casos de prueba, en dónde se evaluó el funcionamiento de los módulos POST, GET ID, GET ALL, PUT, DELETE y sus posibles variaciones.

Los resultados obtenidos denotan un total de 8 casos pasados y 4 fallidos.

Como observación general, las funciones básicas que debería tener una API REST se cumple en la mayoría de los casos, puesto que la conexión entre el proyecto realizado en Java Spring Boot, la base de datos H2 y los endpoints en Postman es estable y no existen errores o interrupciones en la comunicación.

No fue posible obtener una conexión a la funcionalidad del método “PUT”, debido a esto, no se ahondó en más pruebas para esta solicitud hasta que el desarrollador depure esta función.

El resto de endpoints proporcionados por el desarrollador se ejecutan adecuadamente, los path especificados corresponden para cada método del CRUD.

Al momento de modificar los datos de entrada para los atributos de las entidades artista y álbum, se observa que estas guardan los datos en todas las condiciones probadas:

  • Tipo de dato erróneo al esperado
  • Longitud excesiva de caracteres
  • Formato inadecuado para el campo
  • Campos nulos o vacíos
  • Caracteres permitidos
Lo que nos lleva a concluir que no existe una validación de datos desde los atributos, lo cual es importante para asegurarse de que los datos ingresados en el backend cumplan con los requisitos establecidos antes de ser procesados o almacenados en la base de datos.

No existe seguridad para la integridad de los datos, por ende el acceso para la manipulación de datos desde la base de datos es libre para quien sea que tenga el proyecto.

No se estimó necesario realizar pruebas no funcionales como carga y rendimiento, debido al estado prematuro en que se encuentra la aplicación.

Sugerencias:

  • Realizar depuración del código para aquellas pruebas que fueron falladas.
  • Se recomienda validar los datos desde el backend para que los campos sean atingentes al contexto, por ejemplo, si se desea ingresar el número de álbumes, que el campo sólo permita la entrega de datos numéricos.
  • Se recomienda mejorar el formato del archivo README, especificando de forma más detallada los pasos a seguir para ejecutar el proyecto.
  • Como aspecto a mejorar a futuro, considerar la implementación de seguridad para el acceso a la base de datos, como lo son la autenticación y autorización.
